# DataGrid

The **presentational** AG Grid wrapper. Owns rendering, editing, copy/paste, fill, undo/redo, row add/remove, search, theme &#8212; **all in-memory**. Use for **ephemeral** grids (read-once, no persistence).

For persistent entity grids (vendors, items), use the [ConnectedDataGrid](?path=/docs/components-canary-organisms-shared-connected-data-grid--docs) container, which composes this molecule and adds the read source + write/commit pipeline.

**Molecule** &#8212; Canary | Import: `@arda-cards/design-system/canary`

> See the [AG Grid Usage guide](https://arda-cards.github.io/documentation/process/craft/implementation/ag-grid-usage/) on the documentation site for the canonical decision matrix, default settings, and rich-cell recipes.

---

## When to use

<table>
  <thead>
    <tr><th>Use <code>DataGrid</code> when&#8230;</th><th>Reach for <code>ConnectedDataGrid</code> instead when&#8230;</th></tr>
  </thead>
  <tbody>
    <tr>
      <td>The grid is ephemeral &#8212; e.g. a PO item list rendered into a PDF and discarded.</td>
      <td>The grid is an authoritative entity view (vendors, items, suppliers).</td>
    </tr>
    <tr>
      <td>The page already owns persistence and just needs a viewer/editor widget.</td>
      <td>You need the draft &#8594; create &#8594; reconcile lifecycle, or bulk edits to round-trip to the server.</td>
    </tr>
    <tr>
      <td>Refresh is fine resetting all state.</td>
      <td>Edits, adds, and deletes must survive a refresh.</td>
    </tr>
  </tbody>
</table>

When in doubt, choose `ConnectedDataGrid`. Hand-rolling persistence around a bare `DataGrid` is the failure mode this two-tier split exists to prevent.

## What you get by default

Both wrappers register the AG Grid module bundle centrally inside `DataGrid` &#8212; **consumers do not need to register modules**:

```ts
// Done once inside DataGrid, wrapped in try/catch so repeat registration is a no-op
ModuleRegistry.registerModules([AllCommunityModule, AllEnterpriseModule]);
```

This includes the Enterprise Clipboard and CellSelection modules used by the paste / fill / range-selection capabilities below, plus the Enterprise context menus (right-click on a header, row, or cell). An AG Grid Enterprise license key is **not** required for development &#8212; you'll see a watermark and console warning in Storybook, but functionality works.

`DataGrid` hard-codes a few shared grid options:

```ts
{
  stopEditingWhenCellsLoseFocus: true,
  tooltipShowDelay: 400,
  tooltipShowMode: 'whenTruncated',
  ensureDomOrder: true,
}
```

Plus Arda theming (`themeQuartz` with design-system token mapping), focus ring, Enter-to-edit, drag-to-scroll, debounced search, empty/loading overlays, and pinned-right `actionsColumn` support.

## Row editing API (in-memory)

Both wrappers expose the same row-add/remove mechanics on their ref. The container delegates to the molecule:

```ts
ref.addRow(seed?: Partial<T>, opts?: {
  addIndex?: number;            // default 0 (top)
  startEditingField?: keyof T & string;
  history?: boolean;            // default true; pass false for rollback re-inserts
}): string;

ref.removeRows(ids: string[], opts?: { history?: boolean }): void;
ref.undo(): void;
ref.redo(): void;
```

Plus callbacks so a container can react:

```ts
onRowsAdded?:   (p: { rows: T[]; source: 'user' | 'undo' | 'redo' }) => void;
onRowsRemoved?: (p: { rows: T[]; source: 'user' | 'undo' | 'redo' }) => void;
getNewRowId?:   () => string;     // default: `new-${crypto.randomUUID()}`
```

> **Status (2026-06-01).** `addRow` and the events are shipped on the canary branch. **Structural undo/redo of add/remove** (Ctrl+Z removes the just-added row) is designed but **not yet wired** &#8212; track in the add-rows &amp; row-actions design.

A newly-added row defers `startEditingCell` via `setTimeout` past AG Grid's `rowData` reconcile &#8212; without the defer the edit is cancelled. The molecule handles this internally; consumers just call `addRow`.

## Rich cell data types

A rich/editable column &#8212; tokens, address, color, image &#8212; **must** declare a <code>cellDataType</code>. Reason: <code>cellRenderer</code> is cosmetic only; copy/paste/fill/filter/export run against the underlying value via <code>valueFormatter</code> / <code>valueParser</code> / <code>keyCreator</code>. A hand-rolled renderer that returns badges will look right but break all of those silently.

Existing data types ship in <code>canary/molecules/data-grid/cell-data-types/</code>:

<table>
  <thead>
    <tr><th>Type</th><th>Use</th><th>Renderer</th><th>Editor</th></tr>
  </thead>
  <tbody>
    <tr><td><code>roles</code></td><td>Multi-select tokens (e.g. <code>["Vendor", "Carrier"]</code>)</td><td><code>TokenList</code></td><td>Multiselect typeahead with checkboxes</td></tr>
    <tr><td><code>orderMethod</code></td><td>Single token (e.g. <code>"Online"</code>)</td><td><code>TokenList</code> (one badge)</td><td>Single typeahead, <code>clearOnFocus</code></td></tr>
    <tr><td>Address (coming next)</td><td>Composite object &#8212; renders as one line, edits as a multi-field popup.</td><td>Single-line formatter</td><td>Popup form (<code>cellEditorPopup: true</code>)</td></tr>
  </tbody>
</table>

Reference an existing data type by name on the ColDef:

```ts
{ field: 'roles',       headerName: 'Role',         editable: true, cellDataType: 'roles' },
{ field: 'orderMethod', headerName: 'Order Method', editable: true, cellDataType: 'orderMethod' },
```

To create a new one, use the factory pattern (see [AG Grid Usage &gt; Creating a new rich cell data type](https://arda-cards.github.io/documentation/process/craft/implementation/ag-grid-usage/#creating-a-new-rich-cell-data-type)). Don't put a bespoke `cellRenderer` + `cellEditor` on the ColDef.

## Pitfalls

- **Don't register your own AG Grid modules.** They're registered centrally; duplicate registration leads to inconsistent state.
- **Don't reach into `.ag-*` / `.arda-grid-*` selectors in tests.** They're internal and shift with version bumps (this has caused test breakage on the rich-cells refactor).
- **Don't hand-roll `cellRenderer` for an editable rich column.** Use a `cellDataType` factory.
- **Don't wire `onCellValueChanged` *and* an update seam (`onCommit`/`onRowPublish`) without a draft guard.** Auto-publish churn steals cell focus and breaks undo. (Mostly a concern on the `ConnectedDataGrid`, but bare `DataGrid` consumers with their own write logic should mind it too.)
- **Native cell undo (`undoRedoLimit`) does not cover row add/remove.** That's a separate structural undo stack, designed but not yet wired. For now, removing a just-added row requires the consumer to call `ref.removeRows`.
