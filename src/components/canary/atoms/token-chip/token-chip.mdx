# TokenChip

The recipient/token pill shared by chip fields and
`MultiSelectTypeaheadInput` tokens: a neutral rounded-full Badge with a
hover-revealed border, an optional inline action that the chip expands to fit
on hover, and an always-visible × remove button. One component, so recipient
rows and typeahead tokens can never drift apart visually.

The action and × buttons show design-system tooltips, so a
[TooltipProvider](?path=/docs/components-canary-primitives-tooltip--docs)
must be present in the tree (the app provides one at the layout root — same
requirement as Button's `tooltip` prop).

Internal buttons fire on **pointerdown** and shield the event
(`preventDefault` + `stopPropagation`), so host containers that react to
pointerdown — focusing a token, opening a dropdown, double-press editing —
don't also fire.

## TokenChip vs Badge

Use a TokenChip when the pill **is** a value the user placed into a field and
can take back out — a recipient address, a tag being composed. It belongs to
an editing surface, so it carries a × remove button and can host an inline
action.

Use a [Badge](?path=/docs/components-canary-atoms-badge--docs) when the pill
**annotates** something the user reads — a status, category, count, or role —
that is owned and edited elsewhere.

Rule of thumb: if removing it changes *what you're about to submit*, it's a
TokenChip; if it changes *what you're looking at*, it's a Badge being
dismissed.
