# Purchase Order Behaviors

Manages the procurement order lifecycle: creation (from the Order Queue, from
Kanban Cards, from items, free-form, or by copy), line item management,
submission to vendors, acceptance tracking, receiving against lines, and
archiving. The order entity follows a ten-state lifecycle (`NEW` through
`CLOSED`, `CANCELLED`, `REJECTED`, `DISPUTED`).

---

### `PRO::PO::0001` &#8212; Create Purchase Order

  <tbody>

Create a new purchase order from various sources.

- **PRO::PO::0001::0001.UC &#8212; Create PO from Order Queue:** Create a PO
  draft from a vendor group in the Order Queue. Cards transition from
  `REQUESTING` to `REQUESTED` via `accept` events.

- **PRO::PO::0001::0002.UC &#8212; Create PO from Kanban Cards:** Validate card
  compatibility, group by item + UOM, create with aggregated lines. API:
  `POST /v1/order/order/from-kanban-cards`.

- **PRO::PO::0001::0003.UC &#8212; Create Empty Order:** Blank form, persist
  header in `NEW` state. Defaults: `allowPartial=true`, `deliverBy=now+7d`.

- **PRO::PO::0001::0004.UC &#8212; Create Order as Copy:** Duplicate existing
  PO, reset history, clear card attachments, reset item references to current
  versions. **Not yet implemented.**

- **PRO::PO::0001::0005.UC &#8212; Create Order from Header and Lines:**
  Programmatic creation with prescribed data, enriched from item supply.

- **PRO::PO::0001::0006.UC &#8212; Create Order from List of Items:** Select
  items sharing a compatible supplier. Supplier with highest mention count
  across selected items becomes the PO supplier.

- **PRO::PO::0001::0007.UC &#8212; Create Order from Item Identifiers:** Provide
  a list of item entity IDs. One header and one line per item. Bypasses the
  queue for direct ordering.

Browse, filter, sort, and inspect purchase orders across status groupings.

- **PRO::PO::0002::0001.UC &#8212; List POs:** Draft (`NEW`), Receiving
  (`SUBMITTED`/`IN_PROCESS`/`PARTIALLY_RECEIVED`), Completed
  (`RECEIVED`/`CLOSED`/`ARCHIVED`/`CANCELLED`). API:
  `POST /v1/order/order/query`.

- **PRO::PO::0002::0002.UC &#8212; View PO Details:** Three API paths:
  header-only (`GET /v1/order/order/{eId}`), full
  (`GET /v1/order/order/full/{eId}`), lines-only
  (`GET /v1/order/order/lines/{eId}`).

- **PRO::PO::0002::0003.UC &#8212; View PO List and Details (Actor-Goal):**
  Filter by status groupings, click a row for full details including header,
  lines, and serviced Kanban Cards.

- **PRO::PO::0002::0004.UC &#8212; View Order Details (E2E):** From the
  `/order-queue` page, click an order item row to open the Item Details Panel.

### `PRO::PO::0003` &#8212; Edit Purchase Order

<table>
  <tbody>
    <tr><th>Persona</th><td><a href="?path=/docs/docs-personas-david-purchasing-manager--docs">David</a></td></tr>
    <tr><th>Status</th><td>Covered</td></tr>
  </tbody>
</table>

Modify PO header fields, add or remove lines, and manage card associations on
draft orders. Only POs in `NEW` state are editable.

- **PRO::PO::0003::0001.UC &#8212; Add Cards to Existing PO:** Select additional
  cards, validate vendor compatibility, merge or add lines. API:
  `PUT /v1/order/order/from-kanban-cards/{eId}`.

- **PRO::PO::0003::0002.UC &#8212; Edit PO Header and Lines:** Modify vendor
  name, contact, shipping terms, delivery address, line quantities, notes.
  API: `PUT /v1/order/order/{eId}`. System-managed fields (`orderNumber`,
  `status`, `goodsValue`) are read-only.

- **PRO::PO::0003::0003.UC &#8212; Update Order Header Properties:** From Draft
  list, edit header in a local copy, persist on Save.

- **PRO::PO::0003::0004.UC &#8212; Add Lines from Kanban Cards:** Select cards
  from Order Queue, choose &#8220;add to existing order,&#8221; pick from
  `NEW`-state orders sorted most-recently-updated first.

- **PRO::PO::0003::0005.UC &#8212; Remove Line:** API:
  `DELETE /v1/order/order/{eId}/lines/{lineEId}`. Cards previously associated
  only with the removed line return to `REQUESTING` via `shelve` events.

- **PRO::PO::0003::0006.UC &#8212; Add Single Line:** From a Kanban Card (item
  values copied), as a blank line (item typeahead selector), or via bulk CSV
  upload. Unmatched CSV item names produce `INVALID` lines.

### `PRO::PO::0004` &#8212; Submit Purchase Order

Finalize a draft PO and advance it for vendor communication.

- **PRO::PO::0004::0001.UC &#8212; Submit PO:** Validate (vendor name required,
  at least one line with title + quantity), send `start-processing` events to
  cards, advance PO to `SUBMITTED`. API: `PUT /v1/order/order/{eId}/submit`.

- **PRO::PO::0004::0002.UC &#8212; Submit and Release:** Same validation, but
  cards transition directly to `FULFILLED` (skipping receiving). For in-store
  pickups or retroactive record-keeping.

- **PRO::PO::0004::0003.UC &#8212; Submit Order API:** Desired submission
  workflow. **Not yet implemented** &#8212; existing services do not change
  `OrderStatus` or initiate vendor communication.

- **PRO::PO::0004::0004.UC &#8212; Submit a PO (Lifecycle Stub):** Placeholder
  reserving the identifier in the lifecycle taxonomy.

- **PRO::PO::0004::0005.UC &#8212; Send Order via Email (E2E):** Pre-populated
  email panel with vendor email, subject line, and item list. Cards advance to
  `REQUESTED` on send confirmation.

### `PRO::PO::0008` &#8212; Order Data Rules

<table>
  <tbody>
    <tr><th>Persona</th><td><a href="?path=/docs/docs-personas-david-purchasing-manager--docs">David</a></td></tr>
    <tr><th>Status</th><td>Covered</td></tr>
  </tbody>
</table>

Data integrity rules governing line aggregation, validation, vendor
compatibility, numbering, free-form lines, and editability.

- **PRO::PO::0008::0001.FS &#8212; Line Aggregation:** Same item (`eId`,
  fallback to SKU) &#8594; single PO line. Quantity = sum of reorder
  quantities. All contributing cards tracked via `servicedCards`.

- **PRO::PO::0008::0002.FS &#8212; Snapshot Validation at Submit Time:** All
  contributing cards must still be `REQUESTED`. If changed, error shown and
  card list refreshed; header edits preserved.

- **PRO::PO::0008::0003.FS &#8212; Vendor Compatibility:** Single vendor per
  PO. Cards added must have an ItemSupply matching the PO&#8217;s
  `supplierName`.

- **PRO::PO::0008::0004.FS &#8212; PO Number Assignment:** Auto-generated via
  `ScopedSequenceService` (format `PO-000001`) in Full scope; user-entered
  in Lite.

- **PRO::PO::0008::0005.FS &#8212; Free-Form PO Lines:** Not linked to any
  catalog item or card. All fields free text. No card state transitions.

- **PRO::PO::0008::0006.FS &#8212; PO Editability Guard:** Only `NEW` state
  is editable. After submission, the PO is a read-only snapshot. Only
  `privateNotes` and `received` quantities remain writable.

**PRO::PO::0009::0001.FS &#8212; Update Propagation:** When upstream entities
change, open PO fields update automatically: supplier rename &#8594; header
`supplierName`; item name/description &#8594; line `title`/`description`;
ItemSupply price &#8594; line `unitCost`. Applies only to POs in `NEW` state.
Submitted POs are frozen snapshots.
