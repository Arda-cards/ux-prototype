# Consolidated Component Guidelines

> How to create configurable, production-quality Arda UI components.
>
> This is the single reference for building components in any track (stable, canary, extras) of the `ux-prototype` Storybook library. It consolidates and supersedes the per-topic guides listed in the [References](#references) section.

---

## 1. Philosophy

Arda components follow an **existing-library-first** approach, drawing primarily from two upstream sources:

- **shadcn/ui** -- General-purpose UI primitives (buttons, inputs, dialogs, selects, popovers, badges, etc.). Built on Radix UI + Tailwind CSS.
- **AG Grid** -- Data grid and tabular display components. Already a project dependency (`ag-grid-community` + `ag-grid-react` v34). Used for entity grids, sortable/filterable tables, and any component whose core is a row/column data layout.

For each new component, determine which upstream source (if any) is the best starting point, then follow this preference order:

1. **Import and extend** the upstream component when one exists. Add it as a project dependency (or use the existing one), then wrap or extend it with Arda-specific props, tokens, and behavior.
2. **Copy and adapt** the upstream source when the component needs deeper structural changes that wrapping cannot achieve. Copy the source into the component directory, then refactor.
3. **Use as visual/API reference** when the upstream provides a similar but not directly reusable component. Follow its naming, prop conventions, and accessibility patterns.
4. **Build from scratch** only when no upstream analog exists and the component is domain-specific or structurally novel.

Always check the [shadcn/ui component catalog](https://ui.shadcn.com/docs/components) and the [AG Grid component catalog](https://www.ag-grid.com/react-data-grid/) before creating a new component. If an upstream component covers 70%+ of the requirement, prefer options 1 or 2 over building from scratch.

### Core Principles

- **Accessibility first.** Use Radix UI primitives for complex interactive patterns (dialogs, selects, popovers, tooltips). Never hand-roll focus traps, ARIA state management, or keyboard navigation when a Radix primitive exists.
- **Token-driven styling.** All colors, spacing, and typography reference design tokens from `globals.css`. No hardcoded hex values.
- **Type-safe variants.** Use CVA (class-variance-authority) for components with variant dimensions. Export `VariantProps` for consumer type safety.
- **Separated lifecycle interfaces.** Split props by when they change: StaticConfig (design-time), RuntimeConfig (dynamic), InitConfig (mount-time, optional).

## 3. Starting from an Upstream Library

### 3.1 Adding a shadcn Component

When the shadcn catalog has a component that matches your need:

```bash
# Add the shadcn component to the project
npx shadcn@latest add <component-name>

# This places the source in src/components/ui/<component-name>.tsx
```

This gives you a working component with:
- Radix UI primitives for accessibility
- CVA variant definitions
- `cn()` class composition
- `data-slot` attributes for CSS targeting
- TypeScript types via `VariantProps`

### 3.2 Wrapping a shadcn Component (Preferred)

Create an Arda wrapper that imports the shadcn primitive and extends it:

```tsx
import { Button as ShadcnButton, type buttonVariants } from '@/components/ui/button';
import { cva, type VariantProps } from 'class-variance-authority';
import { cn } from '@/lib/utils';

// Arda-specific variant extensions
const ardaButtonVariants = cva(
  'transition-all', // base
  {
    variants: {
      variant: {
        primary: 'bg-primary text-primary-foreground shadow-xs hover:bg-primary/90',
        secondary: 'bg-secondary text-secondary-foreground shadow-xs hover:bg-secondary/80',
        ghost: 'hover:bg-accent hover:text-accent-foreground',
        destructive: 'bg-destructive text-white shadow-xs hover:bg-destructive/90',
      },
      size: {
        sm: 'h-8 rounded-md px-3 text-xs',
        md: 'h-9 rounded-md px-4 text-sm',
        lg: 'h-10 rounded-md px-6 text-sm',
      },
    },
    defaultVariants: { variant: 'primary', size: 'md' },
  }
);

/** Design-time configuration. */
export interface ArdaButtonStaticConfig
  extends VariantProps<typeof ardaButtonVariants> {}

/** Runtime configuration. */
export interface ArdaButtonRuntimeConfig {
  /** Whether the button shows a loading spinner. */
  loading?: boolean;
}

/** Combined props. */
export interface ArdaButtonProps
  extends ArdaButtonStaticConfig,
    ArdaButtonRuntimeConfig,
    React.ButtonHTMLAttributes<HTMLButtonElement> {}

export function ArdaButton({
  variant,
  size,
  loading = false,
  className,
  disabled,
  children,
  ...props
}: ArdaButtonProps) {
  return (
    <button
      className={cn(ardaButtonVariants({ variant, size }), className)}
      disabled={disabled || loading}
      {...props}
    >
      {loading && <Spinner className="size-4 animate-spin" />}
      {children}
    </button>
  );
}
```

### 3.3 Explicit Upstream Library Check Step

Before writing any component code, **always check the upstream catalogs first**:

**A. Check shadcn/ui** (for general UI primitives):

1. Browse [shadcn/ui Components](https://ui.shadcn.com/docs/components) for a matching or similar component.
2. If a match exists at 70%+ coverage:
   - Run `npx shadcn@latest add <component-name>` to add the base component to `src/components/ui/`.
   - Create the Arda wrapper in the target track directory (e.g., `src/components/canary/atoms/<name>/`).
   - Import and extend the shadcn primitive, applying Arda naming, tokens, and interface separation.
3. If a partial match exists (similar pattern but different structure):
   - Copy the shadcn source into the Arda component directory.
   - Refactor: rename, apply Arda conventions, replace styling with Arda tokens.
   - Add a comment noting the shadcn origin URL.

**B. Check AG Grid** (for data grids and tabular layouts):

1. Browse [AG Grid React Features](https://www.ag-grid.com/react-data-grid/) for a matching feature or component pattern.
2. If the component is fundamentally a row/column data display (sortable table, filterable grid, editable cells, row grouping):
   - Use `AgGridReact` from the existing `ag-grid-react` dependency as the rendering core.
   - Create an Arda wrapper that provides Arda-specific `ColDef` defaults, theme tokens, and the StaticConfig/RuntimeConfig interface split.
   - See the existing `createArdaEntityDataGrid` factory in `src/components/extras/organisms/shared/entity-data-grid/` for the established pattern.
3. If only a sub-feature is needed (e.g., cell renderer, status bar panel):
   - Implement it as a custom AG Grid component following the [AG Grid Component docs](https://www.ag-grid.com/react-data-grid/components/).

**C. If no upstream match exists:**

- Check whether the component's interactive pattern has a **Radix UI primitive** (e.g., dialog, popover, tabs) even if shadcn doesn't wrap it the way you need.
- Build from scratch only as a last resort.

This step applies to all tracks (stable, canary, extras) and is part of step 1.4 in the Component Extraction Workflow.


## 4. Component API Design

### 4.1 Interface Separation by Lifecycle Phase

Split component props into distinct TypeScript interfaces based on **when they change**. Only include interfaces that are relevant to the component.

#### StaticConfig (Design Time)

Properties chosen when the component is placed in a layout. These never change after composition.

```tsx
/** Design-time configuration for ArdaWidget. */
export interface ArdaWidgetStaticConfig extends VariantProps<typeof widgetVariants> {
  /* --- View / Layout / Controller --- */
  /** Display label shown above the widget. */
  label?: string;
  /** Accessibility label when no visible label is present. */
  'aria-label'?: string;
}
```

Includes: labels, titles, variant/size, structural variations, accessibility identifiers.

#### RuntimeConfig (Dynamic)

Properties driven by application state that may change during the component's lifetime.

```tsx
/** Runtime configuration for ArdaWidget. */
export interface ArdaWidgetRuntimeConfig {
  /* --- Model / Data Binding --- */
  /** Current value (controlled mode). */
  value?: string;
  /** Called when the value changes. */
  onChange?: (value: string) => void;

  /* --- View / Layout / Controller --- */
  /** Whether the widget is in a loading state. */
  loading?: boolean;
  /** Whether the widget is disabled. */
  disabled?: boolean;
}
```

Includes: dynamic state (loading, disabled, active), event handlers, controlled values, validation errors, conditional rendering flags.

#### InitConfig (Mount Time) -- Optional

Properties that execute once when the component mounts. **Only include when the component actually has mount-time behavior** (async data fetching, one-time computations). Most components do not need this.

```tsx
/** Mount-time configuration for ArdaWidget. */
export interface ArdaWidgetInitConfig {
  /** Fetches initial data when the widget mounts. */
  getInitialData?: () => Promise<WidgetData>;
  /** Initial value for uncontrolled mode. */
  defaultValue?: string;
}
```

#### Combined Props Interface

```tsx
/** Combined props for ArdaWidget. */
export interface ArdaWidgetProps
  extends ArdaWidgetStaticConfig,
    ArdaWidgetRuntimeConfig,
    React.HTMLAttributes<HTMLDivElement> {}
```

Guidelines:
- HTML attribute passthrough (`extends React.HTMLAttributes<...>`) goes on the **combined** interface, not the sub-interfaces.
- Export all sub-interfaces so consumers can use them for partial typing.
- Add JSDoc comments to every interface and every property.
- Purely presentational components (e.g., Badge) may only need `StaticConfig`.
- Omit `InitConfig` if the component has no mount-time behavior.

### 4.2 Model/Data vs. View/Controller Separation

Within each interface, separate props into two semantic groups using JSDoc section comments:

```tsx
export interface ArdaDataGridStaticConfig<T> {
  /* --- Model / Data Binding --- */
  /** Column definitions describing the data shape. */
  columnDefs: ColDef<T>[];

  /* --- View / Layout / Controller --- */
  /** Height of the grid container. */
  height?: string | number;
  /** Additional CSS classes. */
  className?: string;
}
```

**Model/Data Binding** props relate to domain data, state values, data sources, and data callbacks.
**View/Layout/Controller** props relate to visual presentation, layout flags, styling, and UI event handlers.

### 4.3 Controlled/Uncontrolled Pattern

When a component manages a value that can optionally be controlled by a parent:

```tsx
export function ArdaInput({ value: controlledValue, defaultValue, onChange, ...props }: ArdaInputProps) {
  const [internalValue, setInternalValue] = useState(defaultValue ?? '');
  const value = controlledValue ?? internalValue;
  const isControlled = controlledValue !== undefined;

  function handleChange(newValue: string) {
    if (!isControlled) setInternalValue(newValue);
    onChange?.(newValue);
  }

  return <input value={value} onChange={(e) => handleChange(e.target.value)} {...props} />;
}
```

## 5. Variant Management with CVA

Use CVA (class-variance-authority) for all components with variant dimensions. This provides type-safe variant props and matches the shadcn convention.

### 5.1 Defining Variants

```tsx
import { cva, type VariantProps } from 'class-variance-authority';
import { cn } from '@/lib/utils';

const badgeVariants = cva(
  // Base classes (always applied)
  'inline-flex items-center rounded-sm border px-2 py-0.5 text-xs font-medium transition-colors',
  {
    variants: {
      variant: {
        default:     'bg-status-default-bg text-status-default-text border-status-default-border',
        success:     'bg-status-success-bg text-status-success-text border-status-success-border',
        warning:     'bg-status-warning-bg text-status-warning-text border-status-warning-border',
        info:        'bg-status-info-bg text-status-info-text border-status-info-border',
        destructive: 'bg-status-destructive-bg text-status-destructive-text border-status-destructive-border',
        outline:     'bg-transparent text-muted-foreground border-border',
      },
    },
    defaultVariants: {
      variant: 'default',
    },
  }
);

// Export for external composition
export { badgeVariants };
```

### 5.2 Typing with VariantProps

```tsx
export interface ArdaBadgeStaticConfig extends VariantProps<typeof badgeVariants> {
  /** Show a colored dot indicator. */
  dot?: boolean;
}

export interface ArdaBadgeProps
  extends ArdaBadgeStaticConfig,
    React.HTMLAttributes<HTMLDivElement> {}

export function ArdaBadge({ variant, dot, className, children, ...props }: ArdaBadgeProps) {
  return (
    <div className={cn(badgeVariants({ variant }), className)} {...props}>
      {dot && <span className="size-1.5 rounded-full bg-current shrink-0" />}
      {children}
    </div>
  );
}
```

### 5.3 Compound Variants

When variant combinations need special treatment:

```tsx
const buttonVariants = cva('...base...', {
  variants: {
    variant: { primary: '...', ghost: '...' },
    size: { sm: '...', md: '...', lg: '...' },
  },
  compoundVariants: [
    // Ghost buttons at small size get less padding
    { variant: 'ghost', size: 'sm', className: 'px-1.5' },
  ],
  defaultVariants: { variant: 'primary', size: 'md' },
});
```

### 5.4 When CVA is Overkill

For components with zero or one variant dimension and no size axis, a simple conditional with `cn()` suffices:

```tsx
// OK for a simple boolean variant
<div className={cn('base-classes', active && 'bg-primary text-white', className)} />
```

Use CVA when there are **2+ variant dimensions** or **3+ values in a single dimension**.

## 6. Styling

### 6.1 General Rules

1. **Tailwind utility classes** are the primary styling mechanism.
2. **`cn()` from `@/lib/utils`** for all conditional class composition. Never string-concatenate class names manually.
3. **CVA** for variant-driven class selection.
4. **Accept a `className` prop** on every component for consumer overrides.
5. **CSS files** (`.css`) only when Tailwind alone is insufficient (keyframe animations, complex pseudo-element styles). Use `arda-` prefixed class names in CSS files.
6. **Never use inline `style` objects** except for truly dynamic computed values (e.g., `` style={{ width: `${percentage}%` }} ``).

### 6.2 Color Tokens -- No Hardcoded Colors

**Never use hardcoded hex color values.** All colors must reference design tokens defined in `globals.css`.

Correct:
```tsx
className="bg-primary text-foreground border-border"
className="bg-status-success-bg text-status-success-text"
className="bg-[var(--accent-blue)]"   // when no Tailwind class exists
```

Incorrect:
```tsx
className="bg-[#3B82F6] text-[#0A0A0A]"   // hardcoded hex
className="text-gray-500 bg-red-600"        // Tailwind palette (use semantic tokens)
```

### 6.6 Icons

Use **Lucide React** (`lucide-react`). Icons inherit text color.

Preferred (token-based):
```tsx
<Search className="size-[var(--icon-m)]" />    // 20px
<Package className="size-[var(--icon-l)]" />    // 24px
```

Acceptable alternative:
```tsx
<Search className="w-5 h-5" />   // 20px
```

## 9. Component Classification

Use the Atomic Design hierarchy:

### Atoms

Smallest, most fundamental UI building blocks. Standalone, no dependencies on other Arda components.

**Criteria:**
- Contains no domain logic or business rules
- Does not import other Arda atoms, molecules, or organisms
- Generally maps to a single HTML element or a thin wrapper around a Radix primitive
- Highly reusable across different contexts

**Subcategories:** General, Display and Layout, Form, Grid, Other.

### Molecules

Compositions of atoms serving a specific UI pattern. Domain-agnostic.

**Criteria:**
- Composed of multiple atoms or native elements
- Minimal domain knowledge
- Reusable across multiple features or domains

### Organisms

Complex components with business logic, often domain-specific.

**Criteria:**
- Contains domain-specific logic or data structures
- Often connects to APIs or state management
- Composed of multiple molecules and/or atoms

**Sub-types:** Shared organisms (`organisms/shared/`) are configurable across domains. Reference organisms (`organisms/reference/&#123;domain&#125;/`) are tied to a specific business domain.

### Decision Flowchart

1. Does this component import other Arda components? **No** -> Atom. **Yes** -> step 2.
2. Does it contain domain-specific logic? **Yes** -> Organism. **No** -> step 3.
3. Is it reusable across multiple domains? **Yes** -> Molecule. **No** -> Organism.

## Appendix A: What is CVA (Class Variance Authority)?

### Overview

**CVA** ([class-variance-authority](https://cva.style/docs)) is a small TypeScript library for managing CSS class variants in a type-safe way. It is the standard variant management tool used by shadcn/ui and is already a dependency in the `ux-prototype` project.

CVA solves a specific problem: when a component has multiple visual dimensions (variant, size, state) that each map to different sets of CSS classes, the combinatorial logic of selecting the right classes becomes error-prone. CVA provides a declarative, type-safe way to define these mappings.

### The Problem CVA Solves

Without CVA, variant logic typically looks like this:

```tsx
// Manual approach — error-prone, no type safety on variant values
function Badge({ variant = 'default', size = 'md', className }: BadgeProps) {
  const variantClasses: Record<string, string> = {
    default: 'bg-gray-100 text-gray-800',
    success: 'bg-green-100 text-green-800',
    warning: 'bg-yellow-100 text-yellow-800',
  };
  const sizeClasses: Record<string, string> = {
    sm: 'text-xs px-1.5 py-0.5',
    md: 'text-sm px-2 py-0.5',
  };

  return (
    <span className={cn(
      'inline-flex items-center rounded-sm font-medium',
      variantClasses[variant],   // no autocomplete, no compile-time check
      sizeClasses[size],
      className,
    )}>
      ...
    </span>
  );
}
```

Issues: no TypeScript enforcement that `variant` values match the keys, no way to express compound variants (e.g., "ghost + sm needs different padding"), and the variant logic is mixed into the component body.

### How CVA Works

CVA separates variant definitions from the component, and generates both the class-selection function and the TypeScript types:

```tsx
import { cva, type VariantProps } from 'class-variance-authority';

// 1. DEFINE — variant-to-class mapping, separate from the component
const badgeVariants = cva(
  // Base classes (always applied regardless of variant)
  'inline-flex items-center rounded-sm font-medium transition-colors',
  {
    // Variant dimensions — each key is a prop name, each value is a class string
    variants: {
      variant: {
        default: 'bg-status-default-bg text-status-default-text border-status-default-border',
        success: 'bg-status-success-bg text-status-success-text border-status-success-border',
        warning: 'bg-status-warning-bg text-status-warning-text border-status-warning-border',
      },
      size: {
        sm: 'text-xs px-1.5 py-0.5',
        md: 'text-sm px-2 py-0.5',
        lg: 'text-base px-2.5 py-1',
      },
    },
    // Compound variants — special classes for specific combinations
    compoundVariants: [
      { variant: 'default', size: 'sm', className: 'border' },
    ],
    // Defaults when props are not provided
    defaultVariants: {
      variant: 'default',
      size: 'md',
    },
  }
);

// 2. TYPE — extract prop types automatically from the definition
//    VariantProps<typeof badgeVariants> = { variant?: 'default' | 'success' | 'warning'; size?: 'sm' | 'md' | 'lg' }
type BadgeVariantProps = VariantProps<typeof badgeVariants>;

// 3. USE — call the function with variant props to get the resolved class string
function ArdaBadge({ variant, size, className, children }: BadgeVariantProps & { className?: string; children: React.ReactNode }) {
  return (
    <span className={cn(badgeVariants({ variant, size }), className)}>
      {children}
    </span>
  );
}
```

### How It Integrates with Arda Conventions

In Arda components, CVA definitions go at the top of the component file, above the interface definitions:

```tsx
// 1. CVA definition (variant logic)
const widgetVariants = cva('...base...', { variants: { ... }, defaultVariants: { ... } });

// 2. StaticConfig extends VariantProps (variant props are design-time choices)
export interface ArdaWidgetStaticConfig extends VariantProps<typeof widgetVariants> {
  label?: string;
}

// 3. RuntimeConfig (dynamic props, independent of CVA)
export interface ArdaWidgetRuntimeConfig {
  loading?: boolean;
  onChange?: (value: string) => void;
}

// 4. Combined props
export interface ArdaWidgetProps
  extends ArdaWidgetStaticConfig, ArdaWidgetRuntimeConfig, React.HTMLAttributes<HTMLDivElement> {}

// 5. Component uses cn(variants({ ... }), className) for class composition
export function ArdaWidget({ variant, size, loading, className, ...props }: ArdaWidgetProps) {
  return <div className={cn(widgetVariants({ variant, size }), loading && 'opacity-50', className)} {...props} />;
}
```

The pattern is: **CVA owns the variant-to-class mapping, `VariantProps` owns the types, `cn()` merges everything together.**

### When CVA is Not Needed

- Components with **no variant dimensions** (e.g., a simple container or layout wrapper).
- Components with a **single boolean toggle** (just use `cn('base', active && 'active-class')`).
- Components where all styling is **inherited from a wrapped shadcn component** (the shadcn component already has its own CVA definition).
