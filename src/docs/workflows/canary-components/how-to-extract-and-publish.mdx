# How to Create New Migration-Track Components

> A step-by-step guide for extracting reusable components from vendored production code into the migration track. Component source still lives under the historical `canary` package paths.

**Prerequisites:**

- [Consolidated Component Guidelines](consolidated-component-guidelines.md) &#8212; component API design, styling, CVA, Storybook conventions
- [VRT Setup Notes](vrt-setup-notes.md) &#8212; visual regression testing infrastructure and manual inspection procedures
- Storybook running locally (`cd ux-prototype && npm run dev`)

---

## Overview

The migration-track component extraction workflow has four phases:

```
Phase 1: Discovery & Definition
  Identify a pattern -> Define the component concept -> Classify (atom/molecule/organism)

Phase 2: Component Creation
  Check upstream libraries (shadcn/ui, AG Grid) -> Create component + CVA variants -> Stories -> Tests -> MDX docs

Phase 3: Adoption in App/Canary
  Fork import chain -> Replace vendored inline code -> Visual comparison -> Iterate -> Update tests/docs

Phase 4: Publication
  Export from barrel file -> Update package contents docs -> Branch -> PR -> Review -> Merge
```

Each phase is detailed below with exact file paths, commands, and decision criteria.

---

## Phase 1: Discovery &amp; Definition

### Step 1.1: Identify Component

**Goal:** Find a recurring visual, interaction, or coding pattern that is a candidate for abstraction and reuse.

**Sources** (any of the following):

- Vendored production code (`src/vendored/arda-frontend/`)
- Figma mockups ([Arda MVP 2.0](https://www.figma.com/design/JJJ5Yb7t8hMe2LfCX0W5gQ))
- App/Reference Storybook stories
- Prototype stories
- Extras experiments
- New feature designs

**Tell-tale signs:**

- 3+ identical or near-identical code fragments in pages or components
- Similar JSX blocks differing only in label text, data bindings, or variant props
- Repeated Tailwind class patterns across files
- Similar Playwright test assertions or screenshot regions
- Figma component sets with consistent structure across pages

**Example &#8212; DetailFieldRow (C1 from Round 2):**

In `vendored/arda-frontend/components/items/ItemDetailsPanel.tsx` lines 1161-1204, five identical blocks:

```tsx
<div className="flex flex-col gap-1">
  <span className="text-sm text-[#737373] font-medium">SKU</span>
  <span className="text-base text-[#0a0a0a] font-semibold break-all">
    {item.sku || 'No SKU available'}
  </span>
</div>
```

Same pattern appears in other vendored panels (supplier details, order details), with only the label, value, and fallback text varying.

**Output:** A short description of the pattern, where it occurs, and how many instances exist.

### Step 1.3: Classify

Use the decision flowchart from the [Consolidated Component Guidelines, Section 9](consolidated-component-guidelines.md#9-component-classification):

## Phase 2: Component Creation

### Step 2.1: Check Upstream Libraries

**Always do this first** (see [Consolidated Guidelines, Section 3.3](consolidated-component-guidelines.md#33-explicit-upstream-library-check-step)):

**A. Check shadcn/ui** (general UI primitives):

1. Browse [shadcn/ui Components](https://ui.shadcn.com/docs/components) for a match.
2. If a match exists (70%+ coverage): `npx shadcn@latest add <component-name>`, then wrap/extend.
3. If partial match: copy and adapt the shadcn source.

**B. Check AG Grid** (data grids and tabular layouts):

1. Browse [AG Grid React Features](https://www.ag-grid.com/react-data-grid/) for a match.
2. If the component is fundamentally a row/column data display: wrap `AgGridReact` with Arda conventions (see the `createArdaEntityDataGrid` factory in `src/components/extras/organisms/shared/entity-data-grid/`).

**C. If no upstream match exists:** check for a Radix UI primitive, or build from scratch.

**For DetailFieldRow:** No shadcn or AG Grid analog exists. This is a simple display atom &#8212; build from scratch.

### Step 2.2: Create the Component File

**Directory:** `src/components/canary/atoms/{component-name}/`

```bash
mkdir -p src/components/canary/atoms/detail-field
```

**File:** `detail-field.tsx`

Follow the structure from the [Consolidated Guidelines, Section 4](consolidated-component-guidelines.md#4-component-api-design):

```tsx
import { cva, type VariantProps } from 'class-variance-authority';
import { cn } from '@/lib/utils';

// --- CVA variant definition ---

const detailFieldVariants = cva(
  'flex flex-col gap-1', // base
  {
    variants: {
      variant: {
        default: '',
        compact: 'gap-0.5',
      },
    },
    defaultVariants: { variant: 'default' },
  },
);

// --- Interfaces ---

/** Design-time configuration. */
export interface ArdaDetailFieldStaticConfig extends VariantProps<typeof detailFieldVariants> {
  /** The field label displayed above the value. */
  label: string;
  /** Text shown when value is undefined/empty. */
  fallback?: string;
}

/** Runtime configuration. */
export interface ArdaDetailFieldRuntimeConfig {
  /* --- Model / Data Binding --- */
  /** The field value to display. */
  value?: string;
  /** Custom value rendering (overrides value + fallback). */
  children?: React.ReactNode;
}

/** Combined props. */
export interface ArdaDetailFieldProps
  extends
    ArdaDetailFieldStaticConfig,
    ArdaDetailFieldRuntimeConfig,
    React.HTMLAttributes<HTMLDivElement> {}

// --- Component ---

export function ArdaDetailField({
  label,
  value,
  fallback = '\u2014',
  variant,
  children,
  className,
  ...props
}: ArdaDetailFieldProps) {
  const display = children ?? (value || fallback);

  return (
    <div className={cn(detailFieldVariants({ variant }), className)} {...props}>
      <span className="text-sm text-muted-foreground font-medium">{label}</span>
      <span className="text-base text-foreground font-semibold break-all">{display}</span>
    </div>
  );
}

export { detailFieldVariants };
```

**Key checks:**

- [ ] Uses design tokens (`text-muted-foreground`, `text-foreground`), not hardcoded hex
- [ ] Accepts `className` prop for consumer overrides
- [ ] CVA for variants (even if only one dimension for now)
- [ ] `VariantProps` in StaticConfig
- [ ] JSDoc on every interface and property
- [ ] Model/Data vs View/Controller comments in interfaces

## Phase 3: Adoption in App/Canary

### Step 3.1: Identify Replacement Points

Review each App/Canary story for places where the new component should replace inline vendored code.

For DetailFieldRow in the Item Detail App/Canary story:

- The five `flex flex-col gap-1` blocks at lines 1161-1204 of `ItemDetailsPanel.tsx` are the replacement targets.

### Step 3.2: Create the App/Canary Story (if not already present)

If an App/Canary version of the App/Reference story doesn't exist yet, create one.

**Critical rule:** The App/Canary story must render the **same top-level component** as the App/Reference story. If the App/Reference story renders a page component (e.g., `ItemDetailPage`), the App/Canary story must also render a page component &#8212; never a lower-level panel or section directly. This ensures the App/Canary output is visually comparable with the App/Reference original.

Steps:

1. Read the App/Reference story and note which component it imports (the "story root component").
2. Copy the App/Reference story file to `src/canary-refactor/`.
3. Change the title to `App/Canary/{Section}/{Page}`.
4. The `withFullAppProviders` decorator auto-wraps stories with `App/Canary` title prefix (configured in Round 3).
5. Keep the same args shape and play functions. At this point the story should render identically to App/Reference.

**Anti-pattern:** Do NOT change the story to render a different component than the App/Reference version. For example, if App/Reference renders `ItemDetailPage`, the App/Canary must NOT render `ItemDetailsPanel` directly &#8212; that produces a completely different visual output and defeats the purpose of side-by-side comparison.

### Step 3.3: Fork the Import Chain and Replace Inline Code

To modify vendored code without affecting the App/Reference story, fork the **entire import chain** from the story root component down to the component containing the inline pattern.

**The fork chain concept:**

If the App/Reference story renders `PageComponent` which imports `PanelComponent` which contains the inline pattern:

```
App/Reference story
  -> imports PageComponent (from vendored)
    -> imports PanelComponent (from vendored) <- inline pattern here

App/Canary story
  -> imports PageComponent (FORKED)
    -> imports PanelComponent (FORKED) <- migration-track component replaces inline pattern
```

**Steps:**

1. **Identify the chain.** Trace imports from the story root component to the component containing the inline code. List every component in that chain.
2. **Fork the leaf.** Copy the vendored component that contains the inline pattern into `src/canary-refactor/components/`. Replace inline code with the new migration-track component.
3. **Fork each parent in the chain.** For each component between the story root and the leaf, create a fork that is identical to the vendored version except it imports the forked child instead of the vendored child.
4. **Update the story import.** Change the App/Canary story to import the forked story root component.
5. **Verify.** At this point, the App/Canary story should render identically to App/Reference except for the intentional migration-track component changes.

**Example &#8212; DetailFieldRow in ItemDetailsPanel:**

```
App/Reference: item-detail.stories.tsx
  -> imports ItemDetailPage from '@frontend/app/item/[itemId]/page'
    -> imports ItemsPage
      -> imports ItemDetailsPanel <- 5 inline detail field blocks

App/Canary: item-detail.stories.tsx
  -> imports ItemDetailPage from './components/ItemDetailPage'  (FORKED)
    -> imports ItemsPage from './components/ItemsPage'  (FORKED)
      -> imports ItemDetailsPanel from './components/ItemDetailsPanel'  (FORKED)
        <- ArdaDetailField replaces 5 inline blocks
```

Each forked file is identical to the vendored original except for the changed import path pointing to the next fork in the chain.

**Fixing relative imports in forks:** When a vendored file is copied to `canary-refactor/components/`, any **relative imports to sibling vendored files** (e.g., `'./ManageCardsPanel'`) will break because those siblings don't exist in the new directory. Convert these to absolute `@frontend/...` paths pointing back to the original vendored location:

```diff
# In the forked file (canary-refactor/components/ItemDetailsPanel.tsx)
- import { ManageCardsPanel } from './ManageCardsPanel';
+ import { ManageCardsPanel } from '@frontend/components/items/ManageCardsPanel';
```

Only change the import for the **next file in the fork chain** to a relative path (e.g., `'./ItemDetailsPanel'`). All other imports should use `@frontend/...` to resolve back to vendored originals.

**Note:** `@frontend` is a Vite alias configured in `.storybook/main.ts`, not in `tsconfig.json`. TypeScript will show "Cannot find module" errors for `@frontend/...` imports, but Storybook resolves them correctly at runtime.

**Shortcut:** If the story root component directly imports the target component (chain length = 1), only one fork is needed.

**Alternative &#8212; Approach B: Wrapper component.** Create a thin wrapper that renders the vendored component but patches its output (less clean, more fragile). Not recommended.

### Step 3.4: Visual Comparison

Compare the App/Canary output against the App/Reference original. This step is mandatory &#8212; do not skip it.

**Method A: Automated VRT (if baselines exist)**

```bash
npm run test:vrt
```

**Method B: Playwright MCP (recommended for agent workflows)**

If a Playwright MCP server is available, use it to verify that the App/Canary story loads and renders correctly. This catches import errors, runtime exceptions, and structural rendering problems that unit tests cannot detect.

Steps:

1. Navigate to the App/Reference story:
   ```
   browser_navigate: http://localhost:6006/?path=/story/{app-current-story-id}--default
   ```
2. Wait for the story iframe content to load (look for a known heading or element inside the iframe):
   ```js
   const frame = page.frameLocator('iframe#storybook-preview-iframe');
   await frame.getByRole('heading', { name: /ExpectedHeading/i }).waitFor({ timeout: 30000 });
   ```
3. Take a screenshot of the App/Reference version:
   ```
   browser_take_screenshot: scratch/app-current-{page-name}.png
   ```
4. Navigate to the App/Canary story:
   ```
   browser_navigate: http://localhost:6006/?path=/story/{app-migration-story-id}--default
   ```
5. Wait for the iframe content and take a screenshot:
   ```
   browser_take_screenshot: scratch/app-migration-{page-name}.png
   ```
6. Compare the two screenshots. They should be visually identical at the page level. Any differences should be limited to the intentional migration-track component changes.

**Common failures caught by Playwright verification:**

- "Failed to fetch dynamically imported module" &#8212; broken import in a forked file (usually a relative import that wasn't converted to `@frontend/...`)
- Blank iframe &#8212; the story renders but the component tree fails silently
- Different page structure &#8212; the story renders a different component than App/Reference (e.g., panel directly instead of full page)

**Method C: Manual inspection (fallback)**

1. Open Storybook (`npm run dev`)
2. Open `App/Reference/Items/Item Detail/Default` in one browser tab
3. Open `App/Canary/Items/Item Detail/Default` in another tab
4. Compare visually, paying attention to:
   - Layout and spacing (gap sizes, padding)
   - Typography (font size, weight, color &#8212; the vendored code uses hardcoded hex like `#737373`, the migration-track component uses `text-muted-foreground`)
   - Border radius and shadows
   - Interactive states
5. Document any intentional deviations (e.g., "label color now uses design token instead of hardcoded hex")

See [VRT Setup Notes](vrt-setup-notes.md) for detailed manual inspection and Playwright-based comparison procedures.

### Step 3.5: Iterate

Based on the visual comparison:

1. **Adjust the component** if the deviation is unintentional (wrong spacing, wrong font weight).
2. **Accept the deviation** if it's an intentional improvement (using design tokens instead of hardcoded hex).
3. **Document deviations** in the component's MDX file under a "Deviations from Vendored" section.
4. Re-run visual comparison after each adjustment until settled.

### Step 3.6: Extend to Additional Stories

Review other App/Reference stories for the same pattern:

1. Identify App/Reference stories that contain the same inline pattern.
2. If an App/Canary copy doesn't exist, create one (Step 3.2).
3. Apply the component replacement (Step 3.3) and visual comparison (Step 3.4).
4. Update the VRT spec to include the new App/Canary stories.

### Step 3.7: Update Tests and Documentation

1. Add the App/Canary stories to `tests/vrt/full-app-pages.spec.ts` (or a new `app-migration-comparison.spec.ts`).
2. Update the component's MDX documentation with adoption examples.
3. Run `npm run test` to ensure all unit tests pass.

## Phase 4: Publication

### Step 4.1: Export from the Barrel File

Add the new component to the appropriate barrel file so consumers can import it. For canary components, update `src/canary.ts`:

```tsx
// In src/canary.ts, under the appropriate // --- Section --- comment

export { ArdaDetailField, detailFieldVariants } from './components/canary/atoms/detail-field/detail-field';
export type {
  ArdaDetailFieldProps,
  ArdaDetailFieldStaticConfig,
  ArdaDetailFieldRuntimeConfig,
} from './components/canary/atoms/detail-field/detail-field';
```

Then regenerate the package contents inventory:

```bash
node tools/update-package-contents.js
```

See [Publishing &#8594; Pre-publish: Update Barrel Exports and Package Contents Docs](?path=/docs/docs-workflows-publishing--docs) for the full barrel file reference.

### Step 4.2: Create Branch and Commit

```bash
git checkout -b jmpicnic/canary-detail-field
git add src/components/canary/atoms/detail-field/
git add src/canary.ts
git add src/canary-refactor/
git add src/docs/workflows/using-the-design-system.mdx
git add tests/vrt/
git commit -m "Add ArdaDetailField canary atom and App/Canary adoption"
```

### Step 4.3: Create Pull Request

```bash
gh pr create \
  --title "Add ArdaDetailField canary atom" \
  --body "## Summary
- New canary atom: ArdaDetailField (label/value pair for detail views)
- Adopted in `App/Canary/Items/Item Detail` (replaces 5 inline blocks)
- Stories, tests, and MDX documentation included

## Verification
- [ ] Storybook: Components/Canary/Atoms/Detail Field renders correctly
- [ ] `App/Canary/Items/Item Detail` matches `App/Reference` visually
- [ ] Unit tests pass: npm run test
- [ ] VRT comparison: npm run test:vrt (if baselines exist)"
```

### Step 4.4: Review and Merge

- Reviewer checks: component API, styling (tokens not hex), story coverage, test coverage, MDX docs.
- Merge to `main` after approval.

   CRITICAL: The story must render the SAME top-level component as the Dev

   also render a page -- never a lower-level panel or section directly).
2. Trace the import chain from the story root component down to the

   chain into `src/canary-refactor/components/`.

   Each parent fork is identical to the vendored original except it imports
   the forked child instead of the vendored child.
4. Update the App/Canary story to import the forked story root component.
5. IMPORTANT: In every forked file, scan for relative imports (`'./SiblingFile'`).
   Convert any relative import that points to a vendored sibling (NOT part of
   the fork chain) to an absolute `@frontend/...` path. Only the fork-chain
   imports should use relative paths between forked files.
6. Document any intentional deviations (e.g., design token vs hardcoded hex).
