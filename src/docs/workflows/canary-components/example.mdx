## Post-Agent Fix: Broken Relative Imports

The agent correctly forked the import chain (story &#8594; page &#8594; page &#8594; panel) and correctly changed the chain imports. However, the forked files retained **relative imports** to sibling vendored files that no longer exist at the fork location.

**Issue:** When a vendored file is copied to `canary-refactor/components/`, any `./Sibling` imports break because the sibling files are not in the new directory.

**Fix applied manually:** Convert relative sibling imports to absolute `@frontend/...` paths:

In `ItemsPage.tsx` (forked):

```diff
- import { ItemTableAGGrid, type ItemTableAGGridRef } from './ItemTableAGGrid';
+ import { ItemTableAGGrid, type ItemTableAGGridRef } from '@frontend/app/items/ItemTableAGGrid';
```

In `ItemDetailsPanel.tsx` (forked):

```diff
- import { ManageCardsPanel } from './ManageCardsPanel';
- import { ItemCardView } from './ItemCardView';
- import { CardsPreviewModal } from './CardsPreviewModal';
+ import { ManageCardsPanel } from '@frontend/components/items/ManageCardsPanel';
+ import { ItemCardView } from '@frontend/components/items/ItemCardView';
+ import { CardsPreviewModal } from '@frontend/components/items/CardsPreviewModal';
```

**Note:** `@frontend` is a Vite alias configured in `.storybook/main.ts` (not in `tsconfig.json`). TypeScript will show errors for these imports, but Storybook resolves them correctly at runtime.

**Lesson learned:** This is now documented in Step 3.3 of the how-to guide under "Fixing relative imports in forks."

## Observations and Lessons Learned

1. **Fork chain is essential.** The first agent attempt rendered `ItemDetailsPanel` directly in the story (without the fork chain), producing a completely different visual output from App/Current. The instructions were updated to make the fork chain concept explicit and mandatory.

2. **Relative imports break in forks.** When copying a vendored file to a new directory, any `./Sibling` imports must be converted to `@frontend/...` absolute paths. The agent did not catch this automatically.

3. **TypeScript vs. Vite resolution.** The `@frontend` alias is configured only in Storybook's Vite config (`.storybook/main.ts`), not in `tsconfig.json` (which excludes `src/vendored`). Forked files that import via `@frontend/...` will show TypeScript errors in the IDE but resolve correctly at Storybook runtime.

4. **MSW mock data gap.** The story uses `pathname: '/item/mock-item-001'` but this ID doesn't exist in the MSW mock handlers. Both App/Current and App/Migration show the same behavior (item not found, redirects to `/items`). To see the detail panel with the migration-track component, you'd need to click a grid row for an item that exists in the mock data (e.g., `item-001`).

5. **Playwright verification is essential.** Visual comparison via Playwright screenshots caught the broken fork chain issue immediately. This step should be part of every agent's self-check before completing.

6. **Barrel export is a separate step.** The agent created the component and adopted it in the canary refactor, but did not add it to `src/canary.ts`. Without the barrel export, the component cannot be imported by consumers of the published package. This step was performed manually after the agent completed. Future agent prompts should include the barrel export as part of Phase 4.
