# How to Create New Stories

This guide covers the conventions for creating Storybook stories in this project, including CSS theming, Dev Witness page stories, and component stories.

## Intent

This storybook needs to support 2 main goals:

1. Develoment of new Stories and components, using the existing design system in the application.
2. Evolution of the design system itself, including the evolution of components (through the [Canary Components Process](canary-components/how-to-extract-and-publish.mdx)) and the design tokens themselves.

The Current design system is imported from the [Arda frontend application](https://github.com/Arda-cards/arda-frontend-app), [Arda Application Preview](https://arda-mock-demo.vercel.app) in the [vendored](../../vendored/) directory. This is the one that needs to be used for any component or activity that is intended to be part of the final product.

Arda's Design System elements are defined in 3 versions:

1. **Nominal**: These are the components, styles, images, etc. that are expected to be used in any development in the product. They are defined in:
  - [Components](../components/)
  - [Styles](../styles/)
  - [Images](../../../public)

2. **Vendored**: These are the components and styles currently being used in the Arda frontend application. Over time, these and the nominal ones will converge as the existing application adopts the new design system (Nominal).

3. **Canary**: These are the components and styles that are being developed and tested in this project. Over time, these will be merged into the nominal design system.

> <h3>**Warning:**</h3>
>
> **Canary** and **Vendored** artifacts are allowed to depend on `Nominal` artifacts, but not the other way around. No design elements should depend on **Vendored** artifacts.

## Comnponent Usage

A story can choose the components from `Nominal`, `Vendored` or `Canary` using standard typescript imports and is responsible for
resolving any naming conflicts.

## CSS Theming

`Storybook` applies a default theme to all its stories. This theme is the `Nominal` theme, which is an extended copy of the current
style of the `arda-frontend-app` codebase. Over time these may diverge, but eventually the `arda-frontend-app` will simply import the nominal `globals.css` and won't implement its own.

We define two CSS layers (`base` and `theme-override`) to control the cascade. See the [CSS Layer System](?path=/docs/docs-css-layer-system--docs) page for full details on how layers work, which CSS file to import, and known limitations.

### Example: Dev Witness Story with Theme Override

```tsx
// src/dev-witness/<section>/my-page.stories.tsx
import type { Meta, StoryObj } from '@storybook/react-vite';
import MyPage from '@frontend/app/my-page/page';

// Override base theme with vendored app styles
import '@/styles/vendored/globals.css';

const meta: Meta<typeof MyPage> = {
  title: 'Dev Witness/Section/My Page',
  component: MyPage,
  tags: ['app-route:/my-page'],
  parameters: {
    layout: 'fullscreen',
    appRoute: '/my-page',
    appComponent: 'app/my-page/page.tsx',
  },
  args: {
    pathname: '/my-page',
  },
};

export default meta;
type Story = StoryObj<typeof MyPage>;

export const Default: Story = {};
```

### Example: Component Story (No Override Needed)

```tsx
// src/components/extras/atoms/my-atom/my-atom.stories.tsx
import type { Meta, StoryObj } from '@storybook/react-vite';
import { MyAtom } from './my-atom';

// No CSS import needed — uses global default from preview.ts

const meta: Meta<typeof MyAtom> = {
  title: 'Components/Extras/Atoms/My Atom',
  component: MyAtom,
};

export default meta;
type Story = StoryObj<typeof MyAtom>;

export const Default: Story = {};
```
