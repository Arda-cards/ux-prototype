# Using the Design System

How to consume the `@arda-cards/design-system` package in your application, including setup, importing components, applying styles, and choosing between Nominal and Canary versions.

---

## Installation

### 1. Configure the registry

Create or update `.npmrc` in your project root:

```
@arda-cards:registry=https://npm.pkg.github.com
```

### 2. Authenticate

Set a personal access token (PAT) with `read:packages` scope:

```bash
npm login --scope=@arda-cards --registry=https://npm.pkg.github.com
```

### 3. Install the package

```bash
npm install @arda-cards/design-system
```

## Importing Styles

### Nominal theme (default)

Import the base design tokens at the root of your application. This provides all CSS custom properties (colors, spacing, typography) and the Tailwind theme:

```tsx
// app/layout.tsx or similar entry point
import '@arda-cards/design-system/styles';
```

### AG Grid theme

If your application uses AG Grid, import the Arda AG Grid theme alongside the base styles:

```tsx
import '@arda-cards/design-system/styles';
import '@arda-cards/design-system/styles/ag-theme-arda.css';
```

### Canary theme overrides

If you are testing Canary components that require different design tokens, import the Canary CSS **after** the Nominal styles. The Canary CSS uses the `theme-override` CSS layer, so its tokens take precedence regardless of import order:

```tsx
import '@arda-cards/design-system/styles';
import '@arda-cards/design-system/styles/canary/globals.css';
```

See the [CSS Layer System](?path=/docs/docs-css-layer-system--docs) for details on how the `base` and `theme-override` layers interact.

## Importing Components

### Nominal components

Nominal components are production-ready and follow a stable API. Use these for any production development:

```tsx
import { cn } from '@arda-cards/design-system';
```

### Canary components

Canary components are being actively developed and tested. Their API may change between releases. Use them when you want to adopt new components early or contribute feedback:

```tsx
import { TextCellDisplay, NumberCellDisplay, DataGrid } from '@arda-cards/design-system/canary';
```

> **Warning:** Canary component APIs are not guaranteed to be stable across minor versions. Pin your dependency version if you need predictability.

### Extras components

Extras components are off the maturity promotion track. They exist as reference implementations and examples. Use them for rapid prototyping or when no Nominal or Canary equivalent exists:

```tsx
import { ArdaButton, ArdaDataGrid } from '@arda-cards/design-system/extras';
import type { ArdaButtonProps } from '@arda-cards/design-system/extras';
```

> **Note:** Extras components are not actively maintained toward promotion. If you depend on an Extras component in production, consider requesting its extraction into the Canary track.

## Importing Types

Domain model types (entities, value objects, enums) are published from dedicated type barrels. Import types from these paths rather than from the component entry points &#8212; this avoids pulling component runtime code into type-only consumers.

### Extras domain types

The `types/extras` barrel exports all domain types used by the extras-track components:

```tsx
import type {
  BusinessAffiliate,
  Item,
  Money,
  PaginationData,
} from '@arda-cards/design-system/types/extras';
```

### Date/time utilities

Timezone data and duration types are in a separate barrel to avoid bundling the large IANA timezone dataset (~66 KB) when only basic types are needed:

```tsx
import type { IanaTimezoneInfo } from '@arda-cards/design-system/types/extras-date-time';
import {
  COMMON_TIMEZONES,
  searchTimezones,
} from '@arda-cards/design-system/types/extras-date-time';
```

### Nominal and Canary types

The `types` and `types/canary` barrels are currently placeholders. As domain types mature and stabilize, they will be promoted into these barrels following the same tier progression as components.

## Importing Assets

Brand logos, theme indicators, and decorative images are available under the `assets` path:

```tsx
import logo from '@arda-cards/design-system/assets/images/ArdaLogoV1.svg';
import mobileLogo from '@arda-cards/design-system/assets/images/ArdaLogoMobileV1.svg';
```

Canary-specific assets are under the `canary` subdirectory:

```tsx
import canaryLogo from '@arda-cards/design-system/assets/canary/images/ArdaLogoV1.svg';
```

### Typical application setup

Most applications should import only Nominal components and the base styles:

```tsx
// app/layout.tsx
import '@arda-cards/design-system/styles';
import '@arda-cards/design-system/styles/ag-theme-arda.css';
```

```tsx
// app/components/my-page.tsx
import { cn } from '@arda-cards/design-system';
```

### Adopting a Canary component

When a Canary component is ready for early adoption:

1. Import the component from the `canary` path.
2. Optionally import Canary styles if the component requires different tokens.
3. Test thoroughly &#8212; Canary APIs may change.
4. When the component is promoted to Nominal, update the import path (a simple find-and-replace).

```tsx
// Before promotion
import { ArdaDetailField } from '@arda-cards/design-system/canary';

// After promotion to Nominal
import { ArdaDetailField } from '@arda-cards/design-system';
```

## Peer Dependencies

The package expects these as peer dependencies in your application:

- `react` &#8805; 18
- `react-dom` &#8805; 18
- `ag-grid-community` &#8805; 34 &#8212; required by Extras data-grid components
- `ag-grid-react` &#8805; 34 &#8212; required by Extras data-grid components
- `lucide-react` &#8805; 0.400 &#8212; icon library used by multiple components

Bundled dependencies (included in the package, no need to install separately): `class-variance-authority`, `clsx`, `tailwind-merge`.

> **Note:** If you only use Nominal or Canary components that don't reference AG Grid or Lucide, the peer dependency warnings are safe to ignore. Install only what your imports actually require.

## CSS Design Tokens

Components use Tailwind utility classes like `text-foreground`, `text-muted-foreground`, `bg-background`, `border-border`, etc. These classes resolve to CSS custom properties that the consumer must define.

### Option A: Import the full theme

Import `globals.css` to get the complete Arda theme (all tokens, Tailwind base, fonts):

```tsx
import '@arda-cards/design-system/styles';
```

### Option B: Import only the tokens

If you have your own Tailwind setup and only need the color/radius custom properties, import the minimal tokens preset:

```tsx
// Nominal tokens
import '@arda-cards/design-system/styles/tokens.css';

// Or Canary tokens (currently identical; will diverge as Canary evolves)
import '@arda-cards/design-system/styles/canary/tokens.css';
```

This defines the core CSS variables (`--foreground`, `--muted-foreground`, `--background`, `--border`, `--primary`, `--secondary`, `--destructive`, `--accent`, `--card`, `--popover`, `--input`, `--ring`, `--radius`) for both light and dark (`.dark`) modes.
